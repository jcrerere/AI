# LN 城市运行层架构

## 本轮抓取出的新概念

这一轮不是单独补玩法，而是把 LN 从“剧情界面”推向“城市运行层”。

从你的描述里，当前需要正式确立的概念有 6 个：

1. `内置地图`
2. `固定店铺注册`
3. `滚动库存`
4. `代办 / 委托`
5. `待办面板`
6. `低 token 的近景渲染`

这 6 个要一起设计，单做某一个都会发散。

## 1. 内置地图

### 目标

不是做视觉地图，而是做“文字层面的地图感”。

也就是：

- 玩家知道自己在城市中的哪里
- 地点之间有连通关系
- 商店、诊台、赛场、餐厅、地铁站都是地图节点
- 节点不是一次性文字，而是会被记住和复用

这种感觉更接近“指针”：

- 当前地点指向某个节点
- 节点再指向周边节点
- 玩家每次移动都是从一个节点跳到另一个节点

### 现有基础

当前仓库已经有两部分可复用基础：

- [types.ts](d:/tavern_helper_template-main/src/LN/types.ts)
  - 已有 `RegionNode / StreetNode / LocationNode / MapRuntimeData`
- [NodeMapEditor.tsx](d:/tavern_helper_template-main/src/LN/components/right/NodeMapEditor.tsx)
  - 已有地区-街区-地点三层地图结构

所以这层不是从零开始。

### 建议实现

把地图拆成两层：

- `静态骨架`
  - 区域
  - 分区
  - 主干街区
  - 地铁站
  - 大设施锚点
- `动态注册层`
  - 玩家第一次真正接触后生成的店铺
  - 诊台
  - 黑市据点
  - 餐厅
  - 服务场所

静态骨架负责世界稳定性，动态注册层负责生活感和随机性。

### 变量建议

当前已有：

- `world.current_location`
- `world.current_site_type`
- `world.current_site_id`
- `world.current_site_label`
- `mapRuntime`

后续建议新增：

```yaml
world:
  city_runtime:
    current_anchor_id: ""
    current_district_id: ""
    discovered_anchor_ids: []
    discovered_shop_ids: []
```

这样“我在哪”“我发现过什么地方”“我能回哪些地方”就有了稳定锚点。

## 2. 固定店铺注册

### 目标

店铺不是每次都随机成另一家。

第一次进入或第一次真正交互后，这家店就要被写入玩家当前档的城市注册表。

例如：

- 诺丝区·罪吻广场·红绡馆
- 诺丝区·裂帛赛道外环·黑赛下注点
- 艾瑞拉区·某商业环廊·某制服店

### 建议结构

```ts
type RuntimeShopRecord = {
  id: string;
  districtId: string;
  anchorId: string;
  name: string;
  type: 'clothing' | 'chip' | 'clinic' | 'drug' | 'service' | 'restaurant';
  tier: 'street' | 'standard' | 'premium' | 'elite';
  signatureStyle: string[];
  hasBackroom: boolean;
  vendorProfileId: string | null;
  refreshSeed: string;
  firstSeenAt: number;
  loyalty: number;
  discountTier: number;
};
```

### 行为原则

- 店铺名字固定
- 类型固定
- 风格固定
- 档次固定
- 暗柜能力固定
- 货架可更新

也就是说，玩家会记住这家店“是什么店”，而不是每次重来都像陌生商店。

## 3. 滚动库存

### 目标

库存更新要像真实商店，不是“整店重置”。

你已经给了正确方向：

- 新一期是新增，不是全换
- 旧货会保留一部分
- 有些会卖掉或下架
- 风格不要乱变

### 建议实现

库存生成分三层：

1. `主池`
   - 决定这家店常卖什么
2. `滚动层`
   - 决定本期新增什么
3. `惊喜层`
   - 决定少量偏离主池的货

### 刷新策略

每家店保存：

- `refreshSeed`
- `refreshEpoch`
- `soldItemIds`
- `commissionSlots`

生成时按：

- 老库存保留 40%-70%
- 新增 20%-40%
- 惊喜货 1-2 件

这样会有“每次来有变化，但还是那家店”的感觉。

### 服装店特别规则

服装店建议固定这些分类：

- 常服
- 礼服
- 制服
- 夜场装
- 功能装
- 内搭/配件

但系统内部判定不直接显示数值，而是隐藏维度：

- 体面感
- 权威感
- 挑逗感
- 正式感
- 行动便利
- 暴露度
- 舒适度
- 场景适配

玩家看到的是：

- 风格
- 场合建议
- 印象描述
- 品质

### 品质

品质不等于风格。

建议先做：

- 粗制
- 标准
- 精制
- 名牌
- 定制

品质影响：

- 价格
- 面子
- 店主态度
- 是否可触发某些场合反应

## 4. 暗柜与代办

### 暗柜

暗柜不等于“特别好看”。

暗柜的价值来自：

- 不方便公开卖
- 有伪装价值
- 有身份联想
- 有风险
- 来源特殊

例如：

- 官方机构制服
- 仿制身份装
- 限流场所工作服
- 走私礼制服

### 代办 / 委托

这条是为了解决“玩家说一句，商家立刻什么都能搞来”的 AI 迎合问题。

正确结构不是即时购买，而是：

- 玩家提出需求
- 商家判断是否接单
- 如果接单，生成委托
- 收定金 / 给期限 / 标风险
- 到时间后再交付或失败

### 接单判定

商家是否接单，取决于：

- 店铺类型
- 店铺档次
- 是否有暗柜
- 渠道能力
- 玩家熟客度
- 当前区域风险

这意味着：

- 普通服装店不能接夜莺制服代办
- 黑市裁缝能接，但只可能是仿制版
- 暗柜店主能接，但要时间、定金和风险

## 5. 待办面板

### 目标

把“说过的话”和“已经承诺的事情”抓出来，不让它们淹没在正文里。

这层不是传统 RPG 的大任务板，更像：

- 待办
- 行程
- 约定
- 取货提醒

### 适合进入待办的内容

- 定制西服
- 商家代办制服
- 周末约饭
- 到某地取货
- 某天赴约
- 某场黑赛封盘前下注
- 去某条线某个站换乘

### 界面建议

不要做成正文下方提示条。

更好的做法是：

- 固定入口放在 LN 主界面边栏或 `灵能操作` 附近
- 新待办出现时弹轻提示
- 面板入口带红点
- 点开后红点消失

### 字段建议

可以先复用或扩展现有 `active_tasks` 思路，但不要继续沿用旧任务结构。

建议新结构：

```ts
type RuntimeTodo = {
  id: string;
  title: string;
  category: 'commission' | 'meeting' | 'pickup' | 'travel' | 'bet' | 'custom';
  status: 'active' | 'ready' | 'completed' | 'failed' | 'cancelled';
  sourceType: 'shop' | 'npc' | 'system' | 'scene';
  sourceId: string;
  locationLabel: string;
  dueAtMinutes: number | null;
  createdAtMinutes: number;
  summary: string;
  detail: string;
  unread: boolean;
  routeHint: string | null;
};
```

### 创建方式

待办创建应该有两条路：

- `模块确认创建`
  - 下单
  - 预约
  - 取货
  - 地铁换乘
- `AI 结构化辅助创建`
  - AI 输出轻量路由和待办提要
  - 前端确认后入库

不建议完全靠 AI 自由发挥。

## 6. 低 token 的近景渲染

### 核心原则

世界书不保存完整实例，正文不塞完整列表。

### 正文只负责

- 你到了哪里
- 当前场景是什么
- 当前动作入口是什么

### 玩法页负责

- 商品列表
- 赔率
- 地铁线路选项
- 服务菜单
- 店主说明

### 结算后再回写

- 地点
- 账本
- 背包
- 待办
- 极短摘要

这就是 LN 的“近景渲染层”。

## 推荐实现顺序

### 第一步

做 `城市注册表`

- 动态店铺注册
- 地点首次发现
- 当前锚点绑定

### 第二步

做 `服装店`

- 固定店名
- 固定风格
- 滚动库存
- 熟客折扣

### 第三步

做 `代办单`

- 接单
- 定金
- 风险
- 到货时间

### 第四步

做 `待办面板`

- 新事项提示
- 红点
- 面板查看
- 状态流转

### 第五步

把黑赛、餐厅、地铁也挂到同一套架构

这样后面所有城市玩法都会走统一桥接方式。

## 当前判断

这套完全能实现，而且和 LN 现在已有结构并不冲突。

最值的不是先拼命加新玩法，而是先把下面这三层立起来：

- `内置地图 / 城市注册表`
- `固定店铺 + 滚动库存`
- `待办 / 代办面板`

只要这三层立住，后面的购物、黑赛、地铁、约会、诊台、服务场所都会自然接进去。
